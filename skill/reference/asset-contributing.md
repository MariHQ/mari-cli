# Asset: Contributing guide

The on-ramp for new contributors. Convention: GitHub's community-health standards — a
CONTRIBUTING.md that GitHub surfaces on every new issue/PR. Structure check:
`mari asset check <file>`. Scaffold: `mari asset scaffold contributing "<project>"`.

## Canonical structure (required)
- **How to Contribute** — the ways to help and the fastest path to a first contribution.
- **Development Setup** — clone, install, run, test; exact commands.
- **Pull Request Process** — branch, change, test, open a PR; what a good PR looks like.
- **Reporting Bugs** — where/how to file, with repro steps and environment.

Recommended: **Code of Conduct** (link it), **Style Guide**, **Testing**.

## Tone
- Lead with the quickest win — a "good first issue", a docs fix — not policy.
- Link the Code of Conduct rather than restating it.
- Give copy-pasteable commands; assume no prior context about the repo.

## Review rubric
1. All required sections present.
2. Setup commands are exact and actually run from a clean clone.
3. The PR process is concrete (branch from what, checks that must pass).
4. Links to CODE_OF_CONDUCT.md and issue templates resolve.
5. No dead "TODO"/placeholder contacts or commands.

Leans on: structure check + `minimizing-words` ("simply", "just"), `passive-voice`, `vague-link-text`.
